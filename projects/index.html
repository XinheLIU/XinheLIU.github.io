Project note structure

Each project note keeps the problem, boundary, tradeoffs, artifact, and transferable lesson visible.

01 Problem shape

Describe the class of decision problem, not the private organization.

02 System boundary

Explain inputs, outputs, constraints, and feedback signals.

03 Tradeoffs

Speed vs validity, automation vs review, growth vs guardrails.

04 Reusable artifact

Link a diagram, checklist, note, or repo when it is public.

05 Transferable lesson

End with a transferable lesson, not an inflated performance claim.

Publication boundaries

Only public problems, diagrams, methods, and lessons are included here.

boundary

Employer or client identifiers

Replace with domain-level descriptions: enterprise AI, recommendation systems, ads platforms, experiment infrastructure.

system map

Private metrics and internal architecture
